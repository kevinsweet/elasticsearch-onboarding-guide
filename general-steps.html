Steps to Building Search

These are conceptual steps, not a rigid sequence. In practice there's back-and-forth — users revisit decisions, skip ahead, and iterate. The ordering here is roughly chronological for ease of understanding, but a facilitated experience (AI or human) doesn't have to follow this order. Some steps are conditional (e.g., embedding strategy only applies if vectors are involved). Users with more experience will skip steps or make decisions instantly; newer users will loop back as they learn what works.

1
Primary retrieval pattern
The user is answering: "how do people find things in my app?" This shapes everything downstream — query types, mapping, embedding decisions, and what "good results" means.
Text / keyword
Users type words, results ranked by text relevance (BM25). Exact matches, stemming, synonyms. Filters and facets narrow results. Autocomplete suggests as you type.
Vector / semantic
Results ranked by meaning, not exact words. Similarity matching ("find things like this"). Works for text, images, audio — anything that can be embedded. Powers semantic search, RAG, recommendations, deduplication, matching, classification, anomaly detection.
Hybrid
Combines both. The question is which one leads: text-led hybrid uses vectors to catch what keywords miss. Vector-led hybrid uses keywords to fill in exact-match gaps. The balance matters for tuning.
Why this matters early: The retrieval pattern determines which fields need embeddings (Step 2), what query types you'll write (Step 5), and how you tune relevance (Step 7).
How this maps to Elastic's project types
Elastic offers two Serverless project types: ES3 (text-first) and VectorDB (vector-first). There's overlap — the right choice depends on more than just retrieval pattern.

2
Embedding strategy
Conditional — only applies if vectors are involved. Keyword-only users skip this step.
Ingest model = query model: The model you embed with at ingest time must be the same model used at query time. If you change models later, you have to re-embed all your data. This also applies to BYO vectors — whatever generates embeddings for your documents must also generate the query embedding.
semantic_text
Zero-config. Mark the field and Elasticsearch picks a model, chunks long text, generates embeddings, stores everything. You send plain text.
  • Text only
  • Least effort
  • Good for starting fast
  • Chunking configurable (size, overlap) but not custom logic
Fastest start
semantic
Like semantic_text but multi-model. Designed for use with Jina v5 and similar models that produce multiple embedding types from one input.
  • Multiple embedding types from one input
  • Zero-config from the user's side
Multi-model
Inference endpoint + dense_vector
Manual setup: you configure a model connection (inference endpoint) and a vector storage field (dense_vector) separately. More work, full control over model, dimensions, and modality.
  • Jina, OpenAI, Cohere, ELSER, Voyage, Bedrock, Vertex
  • Text, image, video, multimodal
  • Production-grade
  • Any modality
Full control
BYO vectors
Generate embeddings externally and send float[] directly. Any model, any dimension, any modality. This is also the path for self-hosted models (Hugging Face, custom fine-tuned, etc.).
  • For existing pipelines
  • You own the embedding step
  • Just need storage + retrieval
External pipeline
Applies to Inference endpoint + BYO only: Quantization (compress vectors — tradeoff: recall vs cost) and chunking (how to split long documents before embedding). With semantic_text/semantic, these are handled automatically.
3
Mapping
Conditional — can be skipped initially. If you're exploring, load data with no mapping and Elasticsearch auto-detects field types. Come back to define intentional mapping once you know what you're working with.

Mapping decisions

  • Which fields get semantic_text?
  • Which are keyword (facets)?
  • Which need dense_vector?
  • Which are just metadata (long, float, boolean)?
  • Nested vs. flattened for object arrays?
  • Multi-fields (same field as text + keyword)?
  • Analyzer choice (language, synonyms, custom)?
  • Dynamic mapping behavior (strict, runtime, allow new)?
  • Strip unnecessary fields?
  • Data cleanup needed?

Reindex (later)

  • Create index with intentional mapping
  • Reload data from source
  • Generate embeddings (if applicable)
  • Alias swap (zero downtime)
  • Add new mappings when scaling to full dataset
  • Repeat as needed — not a one-time thing
Most users reindex several times as they refine — add vectors, change field types, clean up data. Aliases make each reindex zero-downtime.

Alternative order: Some users flip Steps 2–3 and 4 — they load data first (no mapping) to see what they're working with, then come back to decide embedding strategy and mapping after. This exploratory approach works well when the schema is unknown or messy.
5
Learn to query & test results
Elasticsearch uses its own query language (Query DSL) — a JSON-based format, not SQL or GraphQL. Queries must be tailored to your actual mapping (right field names, right query type per field). This step is about learning the syntax, choosing query types, and confirming your data is searchable.
Learning is part of the work here. Query DSL is proprietary to Elasticsearch — users need to find or be given queries specific to their fields and retrieval pattern. A generic tutorial won't be enough because the queries depend on what was actually mapped and indexed.
No UI yet. You're testing via Dev Tools (Kibana console), cURL, or your client library — hitting the API directly. The frontend comes later (Step 6).
Available query types

Text / keyword

  • Full-text (BM25) — ranked by word relevance
  • Filtered keyword — exact match on categories, IDs, tags
  • Aggregations — counts, averages, histograms over results

Vector / semantic

  • kNN — find nearest vectors by distance
  • Semantic query — send plain text, ES embeds and searches
  • Filtered kNN (ACORN) — vector search within a subset
  • Reranking — re-score initial results with a second model

Hybrid

  • RRF — fuse keyword + vector results by rank
  • Weighted fusion — custom balance between signals
  • Text-led or vector-led — which one dominates
How the query reaches your vectors
  • Built-in inference (semantic_text, semantic) — you send a plain text query like "red running shoes." Elasticsearch embeds it using the same model from ingest, then searches. You never touch vectors directly.
  • BYO vectors — you embed the query yourself in your code (same model you used at ingest), then send the resulting float array to a kNN query.
  • Reranker (optional) — after initial results come back, a second model re-scores them for better relevance. Added on top of either approach above.
Testing checklist
  • Run a sample query — do expected documents come back?
  • Test edge cases — empty queries, very long text, special characters
  • If using vectors — does similarity ranking feel right? Are "close" things actually close?
  • Unexpected results? Note what's off — tune later (don't stop here)

7
Relevance tuning
Iterative loop — not a one-time step
LOOP: tune → eval → tune → eval
Diagnose
  • Semantic matched too broadly?
  • Keyword missed synonyms?
  • Wrong field boosted?
  • Run _rank_eval (P@K, MRR, DCG)
Tune
  • Boosting (weight fields)
  • Reranking (Cohere, cross-encoder)
  • RRF weights
  • Min score threshold
  • Query rules (pin/promote)
  • Synonyms
Measure again
  • Re-run _rank_eval
  • Compare before/after
  • Repeat until satisfied
8
Reindex & data optimization
Most users reindex several times as they refine — add vectors, change field types, clean up data. Aliases make each reindex zero-downtime.
Triggers
  • Mapping was wrong (field types need change)
  • Adding semantic_text to keyword-only fields
  • Switching from semantic_text to dense_vector
  • Data cleanup discovered post-ingest
  • Switching quantization strategy
  • Adding _slice for multi-tenancy
  • Adding new fields from additional sources
Decisions
  • Zero-downtime (alias swap v1 → v2) vs delete
  • Re-run inference on all data? (cost + time)
  • Storage impact (show before/after)
  • Strip Solr artifacts, flatten nested
  • "4,200 fields → which 20 matter?"
  • 10KB/doc → 2KB/doc = 5x savings
9
Production scale & cost
Optimizations for running at scale
Multi-tenancy
Partition vector graphs per tenant (_slice). Thousands of tenants, one index.
Quantization
Compress vectors to reduce memory and cost. Tradeoff: recall vs memory vs cost.
Filtered search (ACORN)
Filtering inside HNSW traversal. Critical for "similar IN this category."
Cost optimization
Unpin idle data, adjust compute scaling, choose the right tier for hot/warm/cold data.
