Skill v1: Use-Case-First Flow

The original 9-step playbook. Asked what you're building first, gathered extensive intent before touching data.

Superseded by v2
1
What are you building?
First question. Use case determines which recipe files load and the overall flow shape.
  • Ecommerce / catalog search
  • RAG chatbot / Q&A
  • Recommendations / similarity
  • Agent memory / context store
  • Log / observability search
  • Enterprise knowledge search
2
Intent questions
Multiple questions asked upfront before any data is touched.
  • Who searches — people in a search bar, or code/API?
  • Natural language queries, exact terms, or both?
  • Do you need facets / filters?
  • Multi-tenant (many customers, one index)?
  • What does "good results" look like for you?
3
Data source
Where the data lives. Same options as v2, but asked after all intent questions.
  • File (JSON, CSV, NDJSON, Parquet, XML, Avro)
  • Database (Postgres, MySQL, MongoDB, DynamoDB, Redis, Cassandra)
  • Cloud storage (S3, GCS, Azure Blob)
  • Data warehouse (Snowflake, BigQuery, Redshift, Databricks)
  • Another search engine (Solr, OpenSearch, Pinecone, Weaviate)
  • SaaS / connectors (Confluence, SharePoint, Google Drive, Salesforce)
  • Streaming / API (Kafka, Kinesis, webhooks, REST APIs)
  • Already in Elasticsearch (reindex, remote reindex, export)
  • No data yet (sample dataset)
4
Language
Programming language for generated code.
  • Python, JavaScript, Java, Go, cURL
5
Ingest
By this point the agent has gathered all context upfront. It designs the mapping AND ingests in one pass.
  • Mapping designed based on earlier intent answers
  • Index created with full mapping immediately
  • Data loaded with proper field types from the start
  • Single pass — no "quick load then re-map"
6
Mapping
Part of step 5 in practice — mapping was designed before ingest based on intent answers.
  • Field types chosen based on use-case answers
  • Embedding strategy selected based on intent
  • No "see the data first, then decide" — decided upfront
7
Search validation
Run a test query to confirm results.
  • Sample query run against indexed data
  • Results shown to user for confirmation
8
Frontend / Pipeline
Build the consuming layer.
  • Search UI for human-facing search
  • LangChain / LlamaIndex for RAG
  • API endpoint for machine consumers
  • Agent Builder integration
9
Iterate
Tune relevance, reindex, scale.
  • Relevance tuning (boosting, reranking, synonyms)
  • _rank_eval for measurement
  • Reindex if mapping changes needed
  • Production scaling (quantization, multi-tenancy)

What changed (v1 → v2)

Testing revealed that users didn't want to answer 5+ upfront questions before seeing anything happen. They wanted data in immediately. The agent also gave better recommendations when it could see actual fields instead of relying on abstract answers about intent.

  • Use-case question moved from step 1 to after ingest — asked once data is visible
  • Intent questions (who searches, natural language vs exact) moved to after ingest
  • "Where's your data?" became the first question — minimum needed to connect
  • Single-pass mapping replaced with two-pass (quick load → mapping decision → reload)
  • All prompts made terse — one question at a time, minimal explanation
  • Write Confirmation Protocol retained (ask before executing)
  • MCP auto-configuration added (no manual cluster setup)
