Why a Checklist Can't Do This

Each "milestone" in a linear checklist hides 4-8 branching decisions underneath. And even the use case alone doesn't determine the right path.

The hidden decision space
A typical milestone path like "RAG context store" looks simple: Create project → Ingest embeddings → Run kNN → Run hybrid with reranking → Connect LLM → Return grounded answer. That's 6 steps. But the actual decision space underneath:
MilestoneReal decisions hidden inside
"Ingest embeddings"semantic_text vs inference endpoint vs BYO? Which model? Which modality? Where's the data? What language? Which fields? Chunk strategy? Cleanup?
"Run kNN"Pure kNN vs hybrid vs semantic? Filtered? Which filter fields? ACORN?
"Run hybrid with reranking"Which reranker? RRF weights? Min score? How many candidates at each stage?
"Connect LLM"Which LLM? LangChain vs LlamaIndex vs direct? Context window? Streaming?
"Return grounded answer"Citations? Source linking? Confidence scores? Fallback behavior?
It goes deeper: even the use case doesn't determine the path
A use-case picker gives you "Catalogue/product search" and routes you to a single project type. But the same use case — ecommerce search — could be ES3 (keyword primary, facets, continuous catalog updates) or VectorDB (semantic discovery primary, scale-to-zero between traffic spikes, multi-tenant marketplace with _slice). The project type depends on write pattern, tenancy needs, and whether vectors are primary or supplemental. You can't know from one card click. A conversation asks 2-3 follow-up questions and resolves it correctly. A static picker has to guess — and will be wrong for a large portion of users.

It assumes a fixed path

A checklist is linear — step 1, 2, 3, done. But developers arrive with completely different starting points. Some have data already in ES. Some have a JSON file. Some have a Postgres database. Some don't even have data yet.

Once it's ingested, the same thing applies. It's not just "run a query" — it's choosing between keyword search, semantic search, hybrid, RAG, vector database patterns, each with different mapping decisions, different inference setups, different query structures. They may drop vectors, they may need to generate them, they may use one model, they may use another. They could be here for searching logs or creating an AI chatbot or keyword search — all very different.

Then after that it's not just "build a frontend" — some users need Search UI, some need an API endpoint, some are feeding results into an AI pipeline that has no frontend at all. Every step fans out into branches that depend on what the user is actually building.

A checklist either flattens all of that into meaningless generics ("Step 3: Search your data") or tries to branch, at which point it's not a checklist anymore — it's a decision tree you're building in UI components instead of in a conversation where it belongs.

To work for everyone, it has to be so generic it says nothing

A checklist that spans all use cases ends up with items like "Ingest your data" and "Run a query" — things the developer already knows they need to do. You're not teaching them anything, you're just restating their own intent back to them as checkboxes.

And the "completion" check is meaningless — did you ingest data? You already know. You were there. You don't need a green checkmark in a sidebar to confirm what you just did. Kibana already has real signals for this — documents show up in Discover, indices appear in Management, search returns results. Those are the actual confirmation moments, and they're already in the product. A checklist widget is a worse version of signals that already exist.

And "run a query" isn't even right — what if it's a retrieval (RAG)? Does Elastic even know when a query is run externally via client library? The checklist can only observe what happens inside Kibana UI, which misses the majority of real usage.

Checklists are finicky

Buggy checking. When do they check? How do they make calls? How do they update? What are the milestones? A "working" checklist needs to observe state across the entire product — indices, documents, queries, pipelines, frontends. That's not a UI widget, it's a state machine that has to observe every possible path through the product.

At the "extremely complex" end: you build a real-time progress tracker that integrates with telemetry, index APIs, and user session state. At the "so vague it's not valuable" end: you build a static list that ignores what's actually happening. There is no good middle ground.

Users probably leave Kibana anyway

Most serious developers build in their IDE. They use Elasticsearch client libraries, write scripts, build applications. A checklist lives in Kibana, but the actual work happens in Cursor, VS Code, or a terminal. By the time a user needs guidance most, they're not looking at the screen where the checklist lives.

It's a distraction from AI-first onboarding

The AI agent already tracks where the user is in the flow — it knows what's been done, what's next, and adapts in real time. Adding a checklist alongside it creates two systems trying to track the same thing. Do they stay in sync? What happens when the AI skips a step because the user's data was already ingested, but the checklist still shows it unchecked?

Now you're building reconciliation logic between two onboarding systems instead of making one good one. And the cost isn't just engineering — it's design and discussion time too. Every checklist item becomes a debate: what's the right label, what counts as "done," does it apply to all solution types, how does it interact with the AI flow. That's weeks of cross-functional alignment on a component that's either redundant (if the AI works) or a crutch (if it doesn't).

We tried this before

In 8.15, a guided onboarding was built where one component was meant to go across all three solutions. It went poorly. Force-fitting, poor UX, failure. That had checklists too. This is a less in-your-face version of the same pattern — trying to create a single linear path through a fundamentally non-linear space. The underlying problem is the same: onboarding is a many-to-many problem, and linear components can't represent it.
