ES3 vs VectorDB

The user is answering: "how do people find things in my app?" The answer determines the project type.

Signals → ES3
  • Users type specific words and expect exact matches
  • Browsing by category, brand, price is the primary navigation
  • Autocomplete / type-ahead is central to the UX
  • Vectors enhance results but don't drive them
  • Data changes constantly — VectorDB's aggressive merging is optimized for "load then search," not concurrent read/write
  • Search runs 24/7 — millions of queries at steady throughput
  • Can't tolerate cold starts — users expect instant response always
Signals → VectorDB
  • Users find things by meaning or similarity, not exact keywords
  • Results feed into an LLM, agent, or AI pipeline
  • Vectors are how the whole system works, not an add-on
  • Data is loaded in batches — corpus doesn't change every minute
  • Some data goes cold over time — idle tenants, old embeddings
  • Cost should scale with how much data is actively searched, not total stored
E-commerce catalog with keyword search, facets, constant inventory updates → ES3
Knowledge base chatbot that embeds docs and retrieves context for an LLM → VectorDB
Product search + semantic_text for "did you mean" style enhancement → debatable (traffic says ES3, Elastic's rule says VectorDB)
What ES3 gives you by default
General-purpose index settings (balanced read/write)
VCU pricing — auto-scaling compute, search always warm (never drops to zero)
No vector-specific optimizations assumed — you add what you need
What VectorDB gives you by default
bfloat16 + DiskBBQ quantization (vectors compressed automatically)
Storage-based pricing — per GB pinned × Search Power. Unpin idle data via Inactivity API to reduce cost.
Aggressive segment merging (faster reads, better recall)
_slice for multi-tenancy (one index, thousands of tenants)
VectorDB Pricing Model (from commercials deck)
Formula: (Pinned GB × Search Power/100 × $10) + (Indexing GB × $2) + (Unpinned GB × $2)
Pinned storage: All data starts pinned (fast, on SSDs) by default. No configuration needed.
Unpinned: Customer-controlled via Inactivity API. Slower queries, lower cost.
Search Power: Performance multiplier, set automatically for most customers (default 100). Advanced mode for power users.
Minimum: $49/month.
Index Mode & _slice Licensing
index.mode: vectordb — available to all tiers (Basic + Standard). DiskBBQ (the performance differentiator) stays Enterprise.
_slice API — available to all tiers. Full enforcement (optimized data placement) is Enterprise self-managed only.
Why: _slice isn't vector-specific — locking the API would harm OSS positioning. Optimized enforcement is where Enterprise earns its differentiation.
ES3 can do everything VectorDB does (kNN, hybrid, quantization) — you just configure it yourself. VectorDB gives you those optimizations out of the box.
