Use Cases ≠ Paths

Knowing what someone is building doesn't tell you how to onboard them. The same use case label can lead to completely different retrieval patterns, embedding strategies, data architectures, and project types — depending on the specifics underneath.

"E-commerce search"
Four teams say they're building e-commerce search. Each ends up on a different path:
Small catalog, keyword-first
  • 500 products, structured fields
  • Facets + filters are the main UX
  • Full-text BM25, no vectors needed
  • ES3, continuous ingest, always-on
Large catalog, semantic discovery
  • 500K products, users describe what they want
  • "Comfy shoes for standing all day"
  • Semantic search primary, filters secondary
  • Vector DB, batch-loaded, semantic_text
Marketplace with multi-tenancy
  • 50 sellers, each with their own catalog
  • Hybrid search (keywords + vectors)
  • _slice per tenant, filtered kNN
  • Vector DB, multi-tenant architecture
Real-time inventory + search
  • Prices and stock change every minute
  • Keyword search + boosted by availability
  • Continuous ingest from streaming source
  • ES3, always-warm, custom scoring
"RAG chatbot"
Three teams building a RAG chatbot. Same label, different everything:
Quick internal Q&A
  • 50 PDF docs, rarely updated
  • semantic_text, zero config
  • Single index, simple semantic query
  • Vector DB, batch load once
Customer support over live data
  • Tickets, docs, and product data — constantly updating
  • Hybrid search (exact order IDs + semantic intent)
  • Multiple indices, continuous ingest
  • ES3, always-on traffic
Multi-model with existing pipeline
  • BYO vectors from Cohere + own reranker
  • Dense vector fields, custom chunking
  • Inference endpoint for reranking stage
  • Either project type works
"Recommendations"
Two teams building recommendations with opposite architectures:
Content recommendations (static corpus)
  • 10K articles, embedded once
  • User reads article → find similar
  • Pure kNN, no text search needed
  • Vector DB, batch load, scale-to-zero friendly
Product recs (live catalog)
  • Products added/removed daily
  • Similarity + filtered by availability/region
  • Filtered kNN + continuous ingest
  • ES3 or Vector DB — depends on update frequency
"Log search"
Same use case, different scale and access patterns:
Dev team debugging
  • A few GB/day of app logs
  • Keyword search + time range filters
  • Short retention, simple mapping
  • ES3, data streams, ILM
Security team investigation
  • Hundreds of GB/day across dozens of sources
  • Correlation queries, long retention
  • Hot/warm/cold tiers, complex pipelines
  • ES3, heavily optimized ingest + storage
The point: A use case label tells you the general domain, but the onboarding path depends on data volume, update frequency, retrieval pattern, existing infrastructure, multi-tenancy needs, cost sensitivity, latency requirements, and expected QPS. This is why onboarding can't be a static flow keyed off "what are you building?" — it needs to ask follow-up questions and adapt based on the answers.
The hidden variables that change everything
Two teams can say "RAG chatbot" and mean completely different operational realities:
Latency
  • Search UI needs <200ms — rules out heavy reranking, limits vector dimensions
  • Background RAG pipeline tolerates 2s — can use reranker, larger models, more candidates
QPS (queries per second)
  • 10 QPS (internal tool) — single shard, minimal infra, cheapest option
  • 10,000 QPS (public-facing) — needs replicas, careful shard strategy, always-warm compute
Throughput (ingest)
  • Load once, rarely update — batch ingest, optimize for read, force-merge
  • Continuous stream of new data — needs concurrent read/write, refresh intervals, ingest pipelines
Cost sensitivity
  • Budget-constrained — quantize aggressively, unpin cold data, scale to zero when idle
  • Performance-first — keep everything in memory, over-provision, pay for speed
None of these are visible from the use case name alone. They only surface when you ask follow-up questions.

User knowledge ≠ consistent either

Even within the same use case and the same path, users arrive with wildly different starting knowledge. This changes how many steps they need help with, which steps they skip, and how much explanation each one requires.

Same project, different people
Five engineers all building the same thing — hybrid search for a product catalog. Each needs a completely different onboarding experience:
Built search on Elasticsearch before
  • Knows Query DSL, mapping, analyzers
  • Skips: language choice, query syntax, basic mapping
  • Needs: what's new (semantic_text, ACORN, RRF), vector-specific setup
  • Loops: minimal — mostly linear
Built search on Solr / OpenSearch
  • Knows search concepts, different syntax
  • Skips: "what is an inverted index" explanations
  • Needs: ES-specific query format, mapping differences, migration cleanup
  • Loops: some — translating mental models
Backend engineer, no search experience
  • Strong at code, APIs, databases
  • Skips: nothing — everything is new
  • Needs: full explanation of retrieval patterns, Query DSL, relevance concepts
  • Loops: many — learn, try, realize it's wrong, redo
Data scientist adding vectors
  • Knows embeddings, models, similarity math
  • Skips: embedding strategy (already decided), chunking, quantization tradeoffs
  • Needs: how to get vectors into ES, kNN query syntax, integration with existing pipeline
  • Loops: few — focused on storage + retrieval mechanics
Developer advocate / prototyper
  • Wants fastest path to a working demo
  • Skips: production concerns, cost optimization, multi-tenancy
  • Needs: semantic_text, sample data, working query in 10 minutes
  • Loops: zero ideally — linear happy path
The implication: Onboarding can't assume a single knowledge level. An experienced Elasticsearch user needs "what's new" — not a tutorial. A search-naive backend engineer needs fundamentals before they can make any decisions. The same step might take 10 seconds for one persona and 30 minutes for another. This is the second reason a static checklist fails — it can't adapt its depth.
